# Инструкция для демонстрации блокчейн-проекта

Проект показывает две идеи:

- как работает простой escrow-контракт с депозитом и выводом средств;
- почему reentrancy-уязвимость опасна и как защита `nonReentrant` + порядок Checks-Effects-Interactions решают проблему.

## 1. Что нужно перед запуском

Нужно установить:

- Node.js;
- зависимости Hardhat;
- браузер.

Если проект запускается с нуля, перейдите в папку с зависимостями и установите их:

```powershell
cd D:\chain
npm install
```

Если PowerShell блокирует `npm` или `npx`, используйте `.cmd`-версии:

```powershell
& 'C:\Program Files\nodejs\npm.cmd' install
& 'C:\Program Files\nodejs\npx.cmd' hardhat compile
```

## 2. Запуск локального блокчейна

Откройте первый терминал:

```powershell
cd D:\chain\blockchain-market
& 'C:\Program Files\nodejs\npx.cmd' hardhat node
```

Этот терминал не закрывать. В нем работает локальная сеть Hardhat на:

```text
http://127.0.0.1:8545
```

Hardhat сразу создает тестовые кошельки с ETH. Это учебные деньги только в локальной сети.

## 3. Демонстрация защищенного контракта

Откройте второй терминал:

```powershell
cd D:\chain\blockchain-market
& 'C:\Program Files\nodejs\npx.cmd' hardhat run scripts/deploy.js --network localhost
```

Скрипт выведет адрес контракта `SecureMarketplace`, например:

```text
Адрес контракта: 0x...
```

Проверьте, что этот адрес стоит в первой строке `app.js`:

```js
var CONTRACT_ADDRESS = "0x...";
```

После этого откройте в браузере:

```text
index.html
```

Что показать:

1. Нажать `Внести 1 ETH`.
2. Показать, что депозит записался в блокчейн.
3. Нажать `Вывести депозит`.
4. Показать, что баланс пользователя стал `0`, а средства вернулись.

Что говорить:

> Это защищенный escrow-контракт. Пользователь отправляет ETH в контракт через `deposit()`, а потом может вернуть их через `withdraw()`. Важный момент: при выводе контракт сначала обнуляет баланс пользователя, а только потом отправляет ETH. Это называется Checks-Effects-Interactions. Еще есть `nonReentrant`, который блокирует повторный вход в функцию во время выполнения.

## 4. Демонстрация уязвимого контракта

В том же втором терминале выполните:

```powershell
& 'C:\Program Files\nodejs\npx.cmd' hardhat run scripts/setup_hack.js --network localhost
```

Скрипт:

- развернет `VulnerableMarket`;
- внесет туда ETH от нескольких пользователей;
- развернет контракт `Attacker`;
- выведет два адреса.

Скопируйте адреса из вывода в `hacker.html`:

```js
const TARGET_MARKET = "адрес VulnerableMarket";
const ATTACKER_CONTRACT = "адрес Attacker";
```

После этого откройте в браузере:

```text
hacker.html
```

Что показать:

1. Обратить внимание на баланс цели.
2. Нажать `Выполнить exploit`.
3. Показать, что баланс уязвимого контракта уменьшается.
4. Объяснить, что атака произошла внутри одной транзакции.

Что говорить:

> Здесь контракт написан неправильно. В `VulnerableMarket.withdraw()` он сначала отправляет ETH пользователю через `call`, а только потом обнуляет его баланс. Если получатель - это не обычный кошелек, а контракт, то его fallback-функция может сразу снова вызвать `withdraw()`. Так получается reentrancy: повторный вход в функцию до завершения первого вызова.

Продолжение:

> Контракт `Attacker` вносит 1 ETH, вызывает `withdraw()`, получает ETH обратно, а в момент получения снова вызывает `withdraw()`. Так он повторяет вывод, пока в уязвимом контракте остаются средства. Это возможно именно потому, что баланс обнуляется слишком поздно.

## 5. Сравнение двух контрактов

Покажите два файла:

```text
contracts\Vulnerable.sol
contracts\SecureMarketplace.sol
```

Главная разница:

Уязвимый вариант:

```solidity
(bool sent, ) = msg.sender.call{value: bal}("");
require(sent, unicode"Ошибка");
balances[msg.sender] = 0;
```

Защищенный вариант:

```solidity
balances[msg.sender] = 0;
(bool sent, ) = msg.sender.call{value: bal}("");
require(sent, unicode"Ошибка перевода");
```

Что говорить:

> В уязвимом варианте внешнее взаимодействие происходит до изменения внутреннего состояния. Это опасно. В защищенном варианте состояние меняется до отправки денег, поэтому повторный вызов уже видит нулевой баланс. Дополнительно используется mutex `nonReentrant`, который запрещает войти в функцию второй раз, пока первый вызов не завершился.

## 6. Короткий план выступления

1. Что такое блокчейн в проекте:
   - локальная сеть Hardhat;
   - блоки;
   - транзакции;
   - кошельки;
   - смарт-контракты.

2. Что делает безопасный контракт:
   - принимает депозит;
   - хранит баланс пользователя;
   - возвращает деньги;
   - защищается от повторного входа.

3. Что делает уязвимый контракт:
   - тоже принимает депозит;
   - тоже выводит деньги;
   - но делает перевод до обновления баланса.

4. Как работает атака:
   - атакующий контракт вносит 1 ETH;
   - вызывает вывод;
   - fallback снова вызывает вывод;
   - цикл повторяется внутри одной транзакции.

5. Как исправляется проблема:
   - сначала менять состояние;
   - потом делать внешний вызов;
   - использовать `nonReentrant`.

## 7. Частые проблемы

Если страница пишет, что не может прочитать контракт:

- проверьте, что запущен `hardhat node`;
- проверьте адрес контракта в `app.js` или `hacker.html`;
- если перезапустили `hardhat node`, заново выполните deploy-скрипты.

Если PowerShell не запускает `npx`:

```powershell
& 'C:\Program Files\nodejs\npx.cmd' hardhat node
```

Если атака не сработала:

- заново запустите `setup_hack.js`;
- скопируйте новые адреса в `hacker.html`;
- обновите страницу браузера.

## 8. Финальная мысль для защиты

> Этот проект показывает, что безопасность смарт-контракта зависит не только от идеи, но и от порядка операций в коде. В блокчейне ошибка особенно опасна, потому что транзакции необратимы. Поэтому для финансовых контрактов важно использовать проверенные паттерны: Checks-Effects-Interactions, reentrancy guard и аварийную паузу.
